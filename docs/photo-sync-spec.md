# 写真同期と通信量制御仕様

この文書は現場共有における写真JPEG同期の正式な安全要件を定める。PR #5では仕様とデータ設計だけを固定し、写真本体アップロードは別ブランチ・別Draft PRで実装する。

## 1. 基本原則

- 撮影写真は通信状態に関係なく、最初に端末内へ保存する。
- Supabase側の保存確認が完了するまで端末内写真を削除しない。
- 初期設定では、モバイル通信や回線種別不明時に自動アップロードしない。
- Webアプリを閉じた後のバックグラウンド同期は保証しない。次回起動時に安全に再開する。
- 現場の切替処理中はアップロードを開始しない。
- クラウド未設定のローカルモードでは同期UIを非表示または無効にする。

## 2. 端末ごとの同期設定

設定値はaoALB固有の端末ローカル設定として保存し、現場間や別端末へ同期しない。

|内部値|画面表示|自動同期|
|---|---|---|
|`wifi_only`|Wi-Fi確認時のみ自動同期|Wi-FiまたはEthernetと明確に判定できた場合だけ|
|`any_network`|すべての回線で自動同期|オンラインなら可能。ただし初回に通信量警告と明示許可が必要|
|`manual`|手動同期のみ|行わない|

初期値と既存利用者の移行後の値は`wifi_only`とする。`any_network`の許可は端末単位で保存し、手動同期の1回限りの許可から自動的に変更しない。

## 3. 回線判定

画面表示は`Wi-Fi`、`モバイル通信`、`回線不明`、`オフライン`の4区分とする。

- `navigator.onLine === false`はオフライン。
- Network Information API等から`wifi`または`ethernet`を明確に取得できた場合だけWi-Fi扱いにする。
- `cellular`はモバイル通信扱いにする。
- API非対応、取得値なし、判別不能は回線不明とする。
- `effectiveType === "4g"`だけでWi-Fiと判断しない。
- `navigator.onLine === true`だけで`wifi_only`の自動同期を開始しない。

iPhone Safari等では回線種別を確実に判定できないため、安全側に`回線不明`として待機する。

## 4. 手動同期

「今すぐ同期」の実行前に、未送信枚数とJPEGの`bytes`合計をKB・MB・GBで表示する。表示値は画像本体を基にした推定で、HTTP等の実通信量は少し増える可能性があることを明記する。

モバイル通信または回線不明時は、枚数と容量を含む確認ダイアログを表示する。利用者の許可はその1回の処理だけに適用する。オフライン時は開始しない。

## 5. 同期キュー

キュー状態は次の5種類とする。

- `pending`: 未送信
- `uploading`: 送信処理中
- `synced`: Supabase側の保存確認済み
- `error`: エラー。利用者操作または安全な再試行対象
- `paused`: 一時停止

各項目は少なくとも`eventId`、`siteId`、`projectUid`、`photoUid`、`sha256`、`bytes`、状態、試行回数、エラー分類、作成・更新・同期日時を保持する。写真Blobはキューへ複製せず、aoALBの`photoFiles`から必要時に読む。

- 同時アップロード数の初期値は1枚。
- `photoUid`とSHA-256で写真の同一性を確認する。
- `eventId`を再送でも維持し、DBの一意制約と合わせて冪等化する。
- 起動時に残った`uploading`は`pending`へ戻す。
- 一時停止時は新しい写真を開始せず、再開時に`pending`から続ける。
- 通信切断、認証切れ、RLS拒否、容量不足、画像不整合を区別して表示する。
- 部分アップロードやSHA-256不一致を`synced`にしない。
- Storage保存とDBの`photo_objects.upload_completed_at`確認後だけ`synced`へ変更する。
- 元のIndexedDB写真とaoPIC由来メタデータは変更しない。

## 6. 進捗UI

次を表示する。

- 未送信枚数・未送信容量
- 同期中、同期済み、エラーの各枚数
- 現在の同期設定と回線判定
- 最終同期日時
- 全体進捗
- 一時停止、再開、エラー写真の再送

同期中に画面を閉じると未送信が残る可能性と、次回起動時に再開することを案内する。「アプリを閉じても必ず送信される」とは表示しない。

## 7. aoALBのダウンロード制御

- 写真一覧で原寸JPEGを一括取得しない。
- 画面付近のサムネイルだけ遅延取得する。
- 原寸は詳細表示や台帳PDF作成など必要時だけ取得する。
- 取得済みBlobはaoALBのIndexedDBへキャッシュし、SHA-256一致時は再取得しない。
- キャッシュ容量を表示し、「端末キャッシュ削除」と「クラウド写真削除」を明確に分ける。
- キャッシュ削除ではクラウド原本を削除しない。

初期版のサムネイルは、追加の有料サービスやサーバー処理を避けるため、アップロード端末で原寸JPEGから生成し、`photoUid`で原寸と対応させる方式を優先する。原寸とは別のStorageパスとSHA-256を持たせ、生成失敗時は原寸保存だけで同期成功扱いにしない。サーバー側生成は、端末負荷・品質統一・Supabase機能利用料を比較して後続段階で再検討する。

## 8. 検証条件

- Wi-Fi、モバイル、回線不明、オフラインの模擬
- `wifi_only`、`any_network`初回警告、`manual`、1回限りの手動許可
- 写真0・1・10・50枚、1枚300～500KB程度の架空JPEG
- 同期中切断、再起動後再開、同じ写真の再送
- 認証切れ、RLS拒否、現場切替、容量不足、破損JPEG
- 元写真・写真メタデータ不変、二重登録なし
- 一覧で原寸一括取得が発生しないこと

## 9. 初期実装の構成

- `js/cloud/network.js`: 回線をWi-Fi・モバイル・不明・オフラインへ安全側に分類する。
- `js/cloud/queue.js`: 既存の`settings`ストアへ端末別設定と写真同期キューを保存する。DBバージョンは変更しない。
- `js/cloud/photo-sync.js`: 送信前のJPEG容量・SHA-256検証と、最大480pxのJPEGサムネイル生成を担当する。
- `js/cloud/supabase-provider.js`: 工事・写真メタデータ、原寸、サムネイル、完了記録、冪等イベントの順に保存する。
- `supabase/migrations/202607190005_photo_thumbnails.sql`: `photo_objects`へサムネイルのパス・SHA-256・bytes・寸法を追加する。PR #5の基盤migration適用後に1回だけ適用する。
- `supabase/rollback/202607190005_photo_thumbnails.sql`: 本migrationだけを戻す必要がある場合のSQL。保存済みサムネイル列を削除するため、実行は対象確認と明示承認後に限る。

クラウド写真の一覧・詳細ダウンロードはこの初期アップロード実装へ含めない。したがって原寸一括取得は追加されず、受信側の遅延取得と端末キャッシュ管理は後続の独立段階とする。
