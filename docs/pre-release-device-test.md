# 公開前 実機試験 手順書（v1 現場共有）

この手順書は、aoPIC（送信側）と aoALB（受信側）の現場共有機能を v1 公開する前に、実機で確認するための手順です。上から順番に実施すると、同じ準備をやり直さずに済みます。専門用語がわからなくても実施できるように書いています。

## 用語のかんたん説明

- 送信側 = aoPIC（写真を撮って送るアプリ）
- 受信側 = aoALB（写真を受け取って台帳を作るアプリ）
- Wi-Fi = 事務所や自宅の無線。モバイル通信 = 携帯回線（4G/5G）。
- キュー = 送信待ちの写真の列。
- Realtime = 新しい写真が届くと受信側へ自動で知らせるしくみ。

## この手順書の使い方

- 各試験の記録表にある「状態」を、次の4つのどれかに更新してください。未実施 / 合格 / 不合格 / 保留。
- 不合格・保留のときは、記録表の「結果メモ・再現手順」に何が起きたかを書き、画面写真（スクリーンショット）を保存してください。
- スクリーンショットのファイル名は「試験番号-連番」で付けてください。例: B6-01, B6-02。
- 一度に全部やらず、依頼された試験を1つずつ実施し、結果を報告してから次へ進んでください。

## 最初に1回だけ行う準備

- 送信スマホ1台（aoPIC用）と受信スマホ1台（aoALB用）を用意します。1台で送受信を切り替えても可能ですが、2台のほうが確認しやすいです。
- 各スマホの標準ブラウザを使います。iPhoneはSafari、AndroidはChrome。
- 検証用の現場（例: AOYAMA_TEST）と、その現場に editor として参加できる参加コードを1つ用意します。
- テスト用の写真を1〜3枚だけ用意します。本番の写真は使いません。

## 優先度の分け方

- 第1段階（公開可否を左右する必須）: これが不合格なら v1 を公開しません。
- 第2段階（できれば公開前に確認）: 不合格でも再送などで復旧できるため、公開を必ずしも止めません。
- 第3段階（公開後の限定運用でも確認可）: サーバー側で担保済み、または影響が限定的なため後回し可能。

## B-9 ローカルモード後方互換

- 分類: 第1段階。理由: 既存の利用者の機能を壊さないことが公開の大前提のため。
- 使用端末: 受信スマホ1台（クラウド設定なし）。
- 回線: Wi-Fi でもモバイルでも可。
- ブラウザ: 標準ブラウザ。

操作手順:

1. クラウド設定をしていない状態で公開URLを開く。
2. 既存のZIPを取り込み、写真一覧・写真詳細を開く。
3. 写真を台帳へ配置する。
4. CSV出力・JSON出力・ZIP出力・黒板合成を1回ずつ試す。
5. 画面を再読み込みして、台帳の配置が元どおり復元されることを確認する。

期待結果: すべて従来どおり動く。クラウド関連のボタンは表示されないか無効。外部への通信が起きない。

合格条件: 既存機能がすべて正常に動き、エラーが出ない。

異常時に保存する情報: どの操作で失敗したか、画面のスクリーンショット、表示されたエラー文言。

## B-6 Realtime到着

- 分類: 第1段階。理由: 現場共有の中核機能で、写真が届かなければ共有が成立しないため。
- 使用端末: 送信スマホ（aoPIC）と受信スマホ（aoALB）の2台。
- 回線: 両方とも Wi-Fi。
- ブラウザ: 標準ブラウザ。

操作手順:

1. 受信側 aoALB を現場に接続し、画面を開いたままにする。
2. 送信側 aoPIC で現場に参加し、権限が editor と表示されることを確認する。
3. 送信側からテスト写真を1枚送信する。
4. 受信側の画面を操作せずに最大1分ほど待つ。

期待結果: 受信側に新着写真が自動で表示される。一覧ではサムネイル（小さい画像）だけが出て、詳細を開いたときに原寸（大きい画像）を取得する。

合格条件: 更新ボタンを押さなくても新着写真が現れる。

異常時に保存する情報: 待った時間、届いたかどうか、受信側の画面のスクリーンショット、送信側の送信済み表示のスクリーンショット。

## B-7 Realtime失敗時の手動更新

- 分類: 第1段階。理由: 中核機能の予備手段。Realtime が不調でも手動で写真を見られないと運用できないため。
- 使用端末: B-6 と同じ2台。
- 回線: Wi-Fi。
- ブラウザ: 標準ブラウザ。

操作手順:

1. 受信側を一度オフライン（機内モードON）にし、数十秒後にオンラインへ戻す。これで自動通知が切れた状況を作る。
2. その間に送信側から写真をもう1枚送信する。
3. 受信側で新着が自動で来ない場合、手動更新ボタンを押す。

期待結果: 手動更新ボタンで新着写真に追いつける。

合格条件: 手動更新のあとに新着写真が表示される。

異常時に保存する情報: 自動で来たか手動で来たか、来ない場合の画面スクリーンショットとエラー文言。

補足: もし「自動でも手動でも来ない」「失敗の表示すら出ない」場合は報告してください。担当が最小のコード追加を検討します（今は変更しません）。

## B-8 実PDF保存（改ページ・文字切れ・画質）

- 分類: 第1段階。理由: 台帳PDFが本アプリの主目的で、崩れると成果物を出せないため。
- 使用端末: 受信スマホ、またはPCでも可。
- 回線: 不問。
- ブラウザ: 標準ブラウザの印刷からPDF保存。

操作手順:

1. 原寸を取得済みの写真を台帳へ3枠配置する。
2. ブラウザの印刷機能からPDF保存を実行する。
3. 保存したPDFを開いて確認する。

期待結果: A4縦・1ページ3枠で改ページが崩れない。台帳の文が枠内に収まる（収まらない場合は印刷が止まる）。写真の画質が実用に耐える。

合格条件: 改ページ・文字切れ・画質のいずれにも問題がない。

異常時に保存する情報: 崩れたページのPDFまたはスクリーンショット、どの枠で文字切れや画質不良が出たか。

## B-5 ブラウザ終了後のキュー復元

- 分類: 第1段階。理由: 送ったつもりが消える・二重送信は信頼性の根幹に関わるため。
- 使用端末: 送信スマホ（aoPIC）。
- 回線: Wi-Fi。
- ブラウザ: 標準ブラウザ。

操作手順:

1. 写真を数枚キューに入れ、送信の途中でブラウザを完全に終了する。
2. もう一度公開URLを開く。

期待結果: 未送信のキューが残っている。送信中で止まっていた分は送信待ちに戻る。同じ写真が二重に登録されない。

合格条件: 写真が消えず、再開でき、重複しない。

異常時に保存する情報: 終了前と再開後の枚数、キュー画面のスクリーンショット。

## B-1 Wi-Fiからモバイル回線への切替

- 分類: 第2段階。理由: 初期設定が Wi-Fi 確認時のみのため影響は限定的で、不具合でも自動送信されない安全側に倒れるため。
- 使用端末: 送信スマホ（aoPIC）。
- 回線: Wi-Fi とモバイルを切り替える。
- ブラウザ: 標準ブラウザ。

操作手順:

1. 設定が Wi-Fi 確認時のみ であることを確認する。Wi-Fi で写真を1枚送り、自動送信されることを確認する。
2. Wi-Fi をOFF（モバイルのみ）にして、写真をもう1枚キューに入れる。

期待結果: モバイルのみでは自動送信されない。画面の回線表示がモバイル通信または回線不明に変わる。

合格条件: モバイル時に勝手に送信されない。

異常時に保存する情報: 切替後の回線表示、送信されたかどうか、スクリーンショット。

## B-2 圏外からの復帰

- 分類: 第2段階。理由: オフライン設計は基盤で検証済みで、復帰後の再送で復旧できるため。
- 使用端末: 送信スマホ（aoPIC）。
- 回線: 機内モードのON/OFFで圏外と復帰を作る。
- ブラウザ: 標準ブラウザ。

操作手順:

1. 機内モードON（圏外）で写真をキューに入れ、送信されないことを確認する。
2. 機内モードOFFにする。
3. 手動同期、またはWi-Fi復帰で送信が再開できることを確認する。

期待結果: 圏外では送信されず、復帰後に送信できる。

合格条件: 圏外で待機し、復帰で再送に成功する。

異常時に保存する情報: 各段階のスクリーンショット、再送できたかどうか。

## B-3 送信中の画面ロック

- 分類: 第2段階。理由: 実機依存だが、失敗しても再送で復旧できるため。
- 使用端末: 送信スマホ（aoPIC）。
- 回線: Wi-Fi。
- ブラウザ: 標準ブラウザ。

操作手順:

1. やや大きめの写真を手動同期する。
2. 送信中に画面をロックし、数十秒後に解除する。

期待結果: 送信が完了する。または未完了なら送信待ち・エラーで残り、再送できる（勝手に消えない）。

合格条件: 写真が消えず、最終的に送信できる。

異常時に保存する情報: ロックした時間、解除後の状態、スクリーンショット。

## B-10 参加コードのレート制限

- 分類: 第3段階。理由: サーバー側での回数制限（15分間に5回失敗で一時ブロック）は実装で確認済みのため、実挙動の確認は公開後の限定運用でも可能。実施すると15分間参加できなくなるので最後に行う。
- 使用端末: 送信スマホ（aoPIC）。
- 回線: Wi-Fi。
- ブラウザ: 標準ブラウザ。

操作手順:

1. わざと誤った参加コードで5回連続、参加を試みる。
2. その後、一時的にブロックされた旨の表示が出ることを確認する。
3. 約15分後に、正しいコードで参加できることを確認する。

期待結果: 5回失敗後に一時ブロックがかかり、15分後に解除される。

合格条件: ブロックがかかり、時間経過後に解除される。

異常時に保存する情報: 何回目でブロックされたか、表示された文言、スクリーンショット。

補足: 本番運用では、参加コードは長めのランダムな文字列を使ってください。総当たり対策として有効です。
